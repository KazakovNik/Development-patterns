# О шаблоне
Основная цель паттерна Абстрактная фабрика (Abstract Factory) — создание семейств взаимосвязанных объектов без привязки к конкретному классу реализации. Вместо простого создания экземпляров конкретных классов используется интерфейс фабрики, который позволяет создавать объекты абстрактного типа, делегируя ответственность за реализацию производящим классам фабрик.

# Основные принципы
Структура компонентов:
- Абстрактная фабрика (AbstractFactory): Интерфейс, определяющий методы для создания продуктов.
- Конкретная фабрика (ConcreteFactory): Реализация интерфейса фабрики, создающая конкретные продукты.
- Абстрактный продукт (AbstractProduct): Общее поведение продукта, представленного разными реализациями.
- Конкретный продукт (ConcreteProduct): Конкретная реализация продукта, создаваемого конкретной фабрикой.
Ключевое отличие заключается в создании семейства связанных объектов, а не отдельных типов, как в паттерне Фабричный метод.

# Преимущества
- Независимость клиентов от реализаций. Клиенты взаимодействуют только с абстрактными типами, что упрощает замену конкретного производителя объектов без - изменения существующего кода.
- Простота расширения системы новыми семьями продуктов. Достаточно создать новую фабрику и реализовать её методы для новых видов продуктов.
- Гибкость сборки сложных структур. Возможность легко комбинировать разные компоненты и создавать целостные группы элементов, работающих согласованно друг с другом.

# Применение в реальных проектах
Пример реального использования паттерна Abstract Factory встречается часто в библиотеках графического интерфейса пользователя (GUI). Например, разработчик создает приложение, которое должно поддерживать разные стили отображения окон и кнопок в зависимости от операционной системы (Windows/MacOS).
Таким образом, каждая система имеет свою специфику визуализации, но общее API остаётся неизменным для всех платформ благодаря абстрагированию через паттерн.
Ещё одним примером является работа с различными базами данных. Одна фабрика создаёт экземпляры драйверов для PostgreSQL, другая — для MySQL, третья — для MongoDB, позволяя приложению динамически переключаться между ними.
